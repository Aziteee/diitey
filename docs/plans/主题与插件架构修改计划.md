# 主题与插件架构修改计划

状态：已实施  
日期：2026-07-13

## 1. 目标

本计划优化主题、插件与核心相遇处的模块形状，不改变现有领域归属：

- 内容文件继续拥有正文与属性；
- 主题继续拥有集合、路由映射、页面与 island；
- 插件继续拥有插件服务、Action、Markdown 扩展与动态数据；
- 核心继续协调三者，维护有效快照和一致的发布视图。

主要目标：

1. 消除主进程与 snapshot worker 分别加载扩展造成的运行上下文不一致风险。
2. 让代码中的 `ContentSnapshot` 回到领域定义：只表示内容记录与索引。
3. 把发布、请求期动态数据、Action、island 和 reload 协调收进一个深 module，缩小核心调用方需要理解的 interface。
4. 删除动态页面使用 `[]` 假数据预渲染以及“分页不能与插件服务组合”等 implementation 限制。
5. 保持现有主题与插件作者 interface，不在本轮引入类型化插件契约或能力系统。

## 2. 不改变的决定

本计划不重开以下 ADR：

- ADR-0001：内容由文件拥有；
- ADR-0002：主题拥有信息架构与页面呈现；
- ADR-0003：插件限于服务端能力与静态 Markdown 扩展；
- ADR-0004：以原子快照发布内容；
- ADR-0005：V1 采用本地单进程部署；
- ADR-0006：主题与插件是可信本地代码；
- ADR-0008：服务启动前自动执行插件迁移。

明确不做：

- 不允许插件自动注入页面、路由映射、CSS、浏览器组件或 island；
- 不加入事件总线、后台任务、定时任务或远程插件协议；
- 不为文件系统、SQLite 或 `Bun.build` 创建公开的远程 port；
- 不改变内容文件与 SQLite 动态数据的所有权；
- 不在内容 `reload` 中重载主题、插件、`site.config.ts` 或执行插件迁移；
- 不在本轮引入多进程部署、Serverless 或插件市场。
- 不修改现有 `PluginDefinition`、`ServiceBinding`、`ActionDefinition` 和 `ValueSchema` 的 public interface；
- 不引入插件契约、能力注册表、服务句柄、Action 句柄或 provider 选择；
- 不改变当前插件服务和 Action 使用字符串名称连接的方式；
- 不拆分 `diitey/theme`、`diitey/plugin`、`diitey/client` 等新包入口；
- 不为插件包增加主题侧 UI 复用协议。

## 3. 目标模块

最终形成以下内部模型：

```text
扩展加载与校验
  → SiteProgram（主题、插件、页面计划、island、programRevision）

内容文件 + SiteProgram
  → ContentPublisher
  → ContentSnapshot（内容记录、集合索引、内容 ID）

SiteProgram + ContentSnapshot
  → EffectivePublication（路由表、静态页面、动态页面计划、资源）

EffectivePublication + 插件动态数据
  → PublicationRuntime（handle / reload / status / close）
```

建议的核心 interface：

```ts
interface PublicationRuntime {
  handle(
    request: Request,
    context?: { readonly clientAddress?: string },
  ): Promise<Response>;

  reload(options?: {
    readonly buildId?: string;
    readonly signal?: AbortSignal;
  }): Promise<ReloadResult>;

  status(): PublicationStatus;
  close(): Promise<void>;
}

async function openPublication(options: {
  readonly root: string;
}): Promise<PublicationRuntime>;
```

`server.ts` 最终只负责公开监听器、管理监听器和进程生命周期，不再直接理解：

- `PublishedPage.dynamic`；
- `ServiceBinding` 的解释规则；
- 插件数据库与插件服务调用参数；
- island manifest 与资源查找；
- 静态页面和动态页面的不同渲染路径；
- reload worker 的替换与快照交换细节。

## 4. 实施阶段

### 阶段 0：锁定现有行为与架构风险

先添加回归测试，不修改生产行为。

新增测试：

1. snapshot worker 超时或异常后，不能在不重启站点的情况下悄悄加载已变化的主题或插件。
2. 使用插件服务的页面可以接收对象、字符串和数字输出，不能假定所有输出都是数组。
3. 一个请求期间的插件服务 `content.exists()` 必须读取该请求捕获的有效快照。
4. 内容 `reload` 不运行插件迁移，也不重新构建 island。
5. 启动失败发生在插件迁移前时，SQLite 不应出现部分变更。

本阶段允许先写出失败测试，用来明确后续阶段的完成条件。

验收：

- 原有 62 个测试保持通过；
- 新测试能稳定复现 worker 上下文分裂和动态页面假数据问题；
- 测试只断言公开行为，不依赖内部函数调用次数或私有对象形状。

建议提交：

```text
test: lock publication runtime invariants
```

### 阶段 1：固定扩展程序，修复 worker 上下文一致性

目标是保证站点启动后，主题、插件和 island 形成一份固定的 `SiteProgram`；内容 `reload` 只能使用这份程序。

修改：

1. 在启动时生成 `programRevision`，记录当前站点配置、已解析主题、插件、页面入口和 island 入口对应的程序版本。
2. 主进程和 snapshot worker 初始化后必须确认使用同一 `programRevision`。
3. worker 返回的每个候选发布结果都携带 `programRevision`；主进程只接受与启动版本相同的结果。
4. worker 超时或崩溃后，第一版采用保守策略：
   - 保留当前有效发布视图；
   - 将 reload 子系统标记为不可用；
   - 后续 reload 明确返回“扩展构建进程不可用，请重启站点”；
   - 不从磁盘静默重新加载主题或插件。
5. 若未来需要自动恢复 worker，应从启动时冻结的程序产物恢复，而不是重新解析当前磁盘上的扩展文件；该优化不进入本阶段。
6. 将 worker 初始化移动到插件迁移之前。主题、插件、页面和 island 的两侧初始化全部成功后，才允许执行 ADR-0008 的数据库迁移。

关键不变量：

- `reload` 前后 `programRevision` 不变；
- 主题、插件或 `site.config.ts` 变化只能通过进程重启生效；
- 任一请求从开始到结束只使用一个有效发布视图；
- worker 失败不能使当前有效发布视图失效。

验收：

- 修改主题后直接执行 reload，不会混用新页面资源与旧插件运行时；
- worker 超时后当前站点继续正常服务；
- worker 初始化失败不会执行插件迁移；
- reload 失败继续保留原有效发布视图。

建议提交：

```text
fix: pin reloads to the startup site program
```

### 阶段 2：纠正内容快照模型

当前代码中的 `ContentSnapshot` 实际包含页面 HTML、动态页面描述和 island 资源，却没有保存完整内容记录。该类型与 `CONTEXT.md` 中的领域含义不一致。

修改：

1. 将真正的内容数据整理为 `ContentSnapshot`：
   - 版本与发布时间；
   - 全部不可变内容记录；
   - 按内容 ID 建立的索引；
   - 集合查询需要的内部索引；
   - 不包含插件动态数据、页面 implementation 或 island 资源。
2. 新增内部 `SiteProgram`：
   - 已校验的主题集合和路由映射；
   - 页面定义和页面 implementation；
   - 插件目录与 Markdown 扩展；
   - island 构建结果；
   - `programRevision`。
3. 新增内部 `EffectivePublication`：
   - `SiteProgram` 与某份 `ContentSnapshot` 的组合结果；
   - 已发布 URL 索引；
   - 纯静态页面结果；
   - 请求期页面计划；
   - 当前可服务的 island 资源。
4. 有效状态只保存一个 `EffectivePublication` 引用；reload 成功后进行一次引用交换。
5. 请求开始时捕获该引用，页面查询、插件服务的内容存在性判断、分页和渲染都从同一引用读取。
6. 更新 `CONTEXT.md`：
   - 保持“内容快照”的既有定义；
   - 为站点可原子服务的整体发布结果补充一个稳定术语。正式命名前先确认使用“有效发布视图”还是其他名称。

文件组织建议：

```text
src/publication/
  site-program.ts
  content-snapshot.ts
  effective-publication.ts
  runtime.ts
```

这不是要求一个 module 只能有一个文件。external interface 保持小，implementation 可以在目录内拆分。

迁移策略：

- 先新增新模型并让旧流程写入新模型；
- 调用方迁移完成后删除旧 `PublishedPage`/`ContentSnapshot` 混合类型；
- 不长期保留新旧两套快照模型和重复测试。

验收：

- `ContentSnapshot` 中能直接读取内容记录，而不是只保存内容 ID；
- `EffectivePublication` 是请求捕获和原子替换的最小单位；
- reload 期间的并发请求仍使用旧的完整有效发布视图；
- 删除旧 module 后，复杂度不会重新散落到 `server.ts` 和插件调用方。

建议提交：

```text
refactor: separate content snapshots from publications
```

### 阶段 3：编译统一的页面数据计划

目标是删除静态页面与动态页面的两套解释流程。

新增内部页面计划：

```ts
interface CompiledPagePlan {
  readonly id: string;
  publish(snapshot: ContentSnapshot): readonly PublishedRouteEntry[];
  render(
    entry: PublishedRouteEntry,
    request: Request,
    runtime: RequestRuntime,
  ): Promise<string>;
}
```

具体形状可以调整，但必须满足：

- 核心只编译一次集合绑定、内容匹配、插件服务绑定和数据依赖；
- 静态页面和动态页面使用同一个页面计划 interface；
- 是否预渲染是 implementation 优化，不改变主题作者 interface；
- `server.ts` 不再重新解释 `ServiceBinding`。

修改：

1. 删除 `emptyServiceData()` 以及用 `[]` 伪造插件服务结果的启动时渲染。
2. 静态页面在发布阶段预渲染；带插件服务的页面在请求期取得真实输出后只渲染一次。
3. 页面计划显式区分：
   - 发布阶段数据：内容记录、集合结果、路由参数和 canonical URL；
   - 请求阶段数据：分页查询参数、插件服务输出和受限请求信息。
4. 插件服务输入引用在编译时解析为内部数据依赖，不再在请求时解释任意字符串路径。
5. 无依赖的插件服务可以并行执行；依赖内容记录字段的插件服务必须在对应发布数据可用后执行。
6. 检测循环依赖和不存在的数据引用，并在启动时报告页面、字段和插件服务名称。
7. 支持分页与插件服务组合：分页先选择当前页的内容记录，再根据页面计划执行需要的插件服务并渲染。
8. 删除 `PublishedPage.dynamic` 和 `renderDynamicPage()`。

验收场景：

- 插件服务输出数组、对象、字符串或数字均可被页面消费；
- 分页页面可以同时读取插件服务；
- 多个无依赖插件服务并行执行；
- 插件服务失败时仍返回统一、无部分 HTML 的 500 页面；
- 静态文章页仍不执行插件服务，也不加载无关 JavaScript。

建议提交：

```text
refactor: compile page data into one execution plan
```

### 阶段 4：深化 PublicationRuntime 并删除旧路径

当快照模型和页面数据计划稳定后，完成核心调用侧收口。本阶段只调整核心 implementation，不修改主题和插件的 public interface。

修改：

1. `openPublication()` 完成以下启动顺序：
   - 加载并校验站点配置、主题与插件；
   - 编译 `SiteProgram`；
   - 构建 island；
   - 构建初始 `ContentSnapshot` 和候选 `EffectivePublication`；
   - 初始化 worker 并确认相同 `programRevision`；
   - 执行插件迁移；
   - 返回可服务的 `PublicationRuntime`。
2. `handle()` 内部统一处理：
   - 主题路由映射；
   - 静态资源；
   - 页面数据计划；
   - 插件服务；
   - Action 安全策略；
   - 页面和 Action 错误归一化。
3. `reload()` 内部负责：
   - 单 reload 并发限制；
   - 超时与取消；
   - 候选内容快照与发布视图构建；
   - `programRevision` 校验；
   - 原子替换与失败保留。
4. `status()` 返回有效发布视图版本、最后成功时间、最近尝试结果和 reload 状态。
5. `close()` 统一关闭 worker、SQLite 和其他内部资源；重复调用无副作用。
6. `server.ts` 只把请求交给 runtime，并保留公开监听器与 loopback 管理监听器的网络隔离。
7. 页面、Action 和 island 资源建立 Map 索引，替代每次请求的线性 `.find()`。
8. 删除旧的浅 module 调用路径和已经被新 interface 测试覆盖的旧内部测试。

测试策略：

- 主要行为测试通过 `PublicationRuntime` interface 运行；
- 文件系统、动态 import、`Bun.build` 和 SQLite 都属于 local-substitutable dependency，只保留内部 seam；
- SQLite 测试优先使用 `:memory:` 或临时数据库，不额外抽象 repository port；
- 保留少量真实 `Bun.build` 和真实动态导入验收测试；
- 测试观察 Response、reload 结果、状态和持久化结果，不测试内部调用顺序。

验收：

- 删除 `snapshot.ts`、`server.ts` 或 `plugins.ts` 中任一旧协调函数时，复杂度不会重新出现在多个调用方；
- 核心调用方只学习 runtime 的少量方法即可获得完整站点行为；
- 所有现有端到端行为继续通过；
- 新 interface 的测试可以承受内部文件重新拆分而无需修改。

建议提交：

```text
refactor: deepen the publication runtime
```

## 5. 风险与控制

| 风险 | 影响 | 控制方式 |
|---|---|---|
| worker 与主进程使用不同扩展版本 | 页面、插件运行时和 island 资源混用 | 阶段 1 优先完成；固定 `programRevision`，失败后要求重启 |
| 动态页面全部请求时渲染 | CPU 成本上升 | 静态页面继续预渲染；动态页面只渲染一次；后续可在 module 内缓存 |
| 阶段图逐渐成为任意 router | 主题绕过核心路由保证 | 主题只能声明路由映射；请求阶段只准备数据，不能认领 URL |
| 新旧模型长期共存 | 重复逻辑与测试增加 | 每阶段明确删除旧路径，采用 replace-don't-layer |
| 内部重构意外改变插件行为 | 主题或插件需要无计划迁移 | 保持 public interface 与 Action URL 不变，增加端到端兼容测试 |

## 6. 每阶段统一验证

每个阶段完成后执行：

```text
bun run typecheck
bun test
git diff --check
```

并人工确认：

- 工作区没有生成的 SQLite、island 临时目录或 runtime info 残留；
- 主题、插件和 `site.config.ts` 的变化仍要求重启；
- 内容 reload 不执行迁移；
- 公开站点不暴露 `/_system/*`；
- Action 的请求体限制、Origin/CSRF、限流、超时和标准错误保持有效；
- 普通静态文章不加载整站 JavaScript；
- 失败构建不替换有效发布视图。

## 7. 推荐执行顺序摘要

```text
阶段 0  回归测试
  ↓
阶段 1  固定 SiteProgram 与 worker 一致性
  ↓
阶段 2  拆分 ContentSnapshot / EffectivePublication
  ↓
阶段 3  统一页面数据计划
  ↓
阶段 4  PublicationRuntime 收口并删除旧路径
  ↓
阶段 5  文档与示例收口
```

前三阶段先修复运行一致性、领域模型和页面执行流程；第四阶段完成核心调用侧收口；最后更新文档与示例。整个计划保持现有主题和插件 public interface，不把 worker 修复、模型拆分和 runtime 收口塞进同一个提交。
