# Diitey - 文件优先个人博客系统：V1 总方案

## 1. 定位

这是一个运行中的个人站点系统，不是静态站点生成器，也不是传统后台 CMS。

核心原则：

- 内容真相永远是 `content/` 中的 Markdown 文件。
- 文件夹就是信息架构的原材料。
- 主题决定如何把文件解释成集合、URL 与页面。
- 插件主要提供后端服务，也可以通过声明式 Markdown 扩展点增加静态内容块渲染；不提供页面布局或交互 UI。
- 动态数据可写入 SQLite；文章正文与属性不复制进数据库。
- 目标是单站点、单 Bun 进程、持久化本地磁盘与 SQLite，不考虑 Serverless、多实例、SEO 重定向。
- 主题与插件都是完全可信的本地代码。后文对它们的职责限制属于受支持接口的约束，不是进程级安全隔离；绕过接口直接读取文件、数据库或导入彼此实现属于未定义行为。

## 2. 技术栈

| 层 | 选择 |
|---|---|
| 运行时、HTTP、CLI、构建 | Bun |
| HTTP 入口 | `Bun.serve` |
| 页面渲染 | Preact |
| 服务端渲染 | `preact-render-to-string` |
| 局部浏览器交互 | Preact islands + `Bun.build` |
| 内容格式 | Markdown + YAML Front Matter |
| Markdown 解析 | `unified` / `remark` / `rehype` |
| 动态数据 | SQLite |
| 内容索引 | 内存中的不可变快照 |

### 术语

- **内容文件**：`content/` 下的 Markdown 源文件，是文章正文与属性的唯一真相来源。
- **内容记录**：核心从一个内容文件解析出的结构化结果，包含稳定身份、源路径、属性与渲染结果。
- **内容快照**：一次完整构建产生的、只读的全部内容记录与索引。线上请求只读取当前有效快照。
- **集合**：主题声明的内容记录查询，不拥有或复制内容。
- **路由映射**：主题声明的 URL 模式与内容源路径模式之间的确定映射。
- **插件服务**：只在服务端调用的类型化插件能力。
- **Action**：浏览器通过核心 HTTP 入口调用插件服务的受控接口。
- **island**：由主题拥有、单独打包并在浏览器 hydration 的交互组件。

## 3. 项目结构

```text
my-site/
  package.json
  bun.lock
  content/
    articles/
      hello.md
    essays.md
    notes/
  themes/
    my-theme/
      theme.ts
      pages/
      islands/
      public/
  plugins/
    comments/
      plugin.ts
      migrations/
  data/
    site.sqlite
  site.config.ts
```

只有 `content/` 会被核心视为可发布内容。`README.md`、插件文档等项目文件不会被误解析。

## 4. 内容模型

每个内容文件必须含有：

```md
---
id: 01JXYZ...
created: "2026-07-12"
title: 我的第一篇文章
tags:
  - blog
draft: false
---

正文……
```

核心强制：

- `id`：全站唯一的稳定身份；文件移动、改名、主题更换都不改变它。V1 不强制具体格式，建议使用 ULID。
- `created`：ISO 8601 日期或带时区时间，用于稳定排序。

V1 的 Front Matter 规范：

- `id` 必须是 YAML 字符串；`created` 必须是 YAML 标量。核心配置 YAML 解析器保留其原始词法值，不把未加引号的日期自动转换为运行时 `Date` 对象。
- `created` 在进入内容记录时规范化为 ISO 8601 字符串。只有日期时按该日处理，不隐式附加本地时区。
- 重复 YAML 键、无法解析的 YAML、空 `id`、无效日期都会使快照构建失败。
- 核心保留 `id` 与 `created`；其他未知属性允许存在，由主题 schema 决定是否接受和如何解释。
- 内容文件与路径按 UTF-8 和 `/` 分隔符规范化；同一规范化路径只允许对应一个内容文件。
- V1 将内容视为站点所有者完全可信，允许 Markdown 中的原始 HTML，不提供面向不可信作者的 HTML 消毒保证。

构建内容快照时，核心必须检查 `id` 的全局唯一性。发现重复 `id` 时，本次构建失败；热重载场景下继续使用上一份有效快照。

主题决定：

- `title`、`tags`、`draft` 等是否必填；
- 自定义属性，如 `status`、`series`、`rating`；
- 哪些路径属于哪种内容。

V1 仅兼容 Obsidian 的 YAML Properties，不支持 `[[双链]]`、`![[嵌入]]`、块引用等 Obsidian 特有语法。

## 5. 主题：信息架构与渲染层

主题是本地、完全可信的 TSX 包。它通过核心提供的主题接口负责：

- 内容集合定义；
- URL 路由；
- 页面布局；
- Preact TSX 页面与 islands。

V1 不对主题做沙箱或能力隔离。主题能够访问 Bun 进程拥有的系统资源；“主题负责信息架构与渲染”是架构约定，不是安全边界。站点只应加载用户明确选择的本地主题。核心只保证通过主题接口提供的数据与插件服务契约稳定；主题直接读取 SQLite、导入插件私有实现或修改内容文件均属于未定义行为。

主题以声明式集合描述内容：

```ts
export default defineTheme({
  collections: {
    articles: collection({
      from: "articles/*.md",
      where: { draft: { not: true } },
      orderBy: [{ field: "created", direction: "desc" }],
      schema: {
        title: "string",
        tags: "string[]?",
        draft: "boolean?",
      },
    }),
  },

  routes: [
    route("/", page("home", {
      recent: { collection: "articles", limit: 10 },
    })),

    route("/writing", page("article-list", {
      items: { collection: "articles", paginate: 20 },
    })),

    route("/writing/:slug", page("article", {
      item: {
        collection: "articles",
        match: "articles/:slug.md",
      },
    })),
  ],
})
```

集合的 `from` 使用 picomatch glob，匹配相对于 `content/` 的规范化源路径。正式支持 `*`、`**`、`?`、字符集合和 brace patterns（例如 `articles/**/*.{md,mdx}`）；主题声明和操作系统路径中的 `\\` 都会先规范化为 `/`。glob 在主题加载时预编译并复用于整次内容快照构建，无效 glob 会直接使启动失败。

受支持的主题页面接口只消费核心注入的数据：

```tsx
export function ArticleList({ items }: Props) {
  return items.map((item) => <a href={item.url}>{item.title}</a>)
}
```

URL 属于当前主题的解释结果。更换主题或移动文件不维护旧 URL，也不做重定向。

### 路由映射规则

- `match` 始终匹配相对于 `content/` 的规范化源路径，不匹配操作系统绝对路径。
- V1 的 `:param` 只匹配一个路径段；嵌套目录必须由主题显式声明多个参数，例如 `articles/:year/:slug.md` 对应 `/writing/:year/:slug`。
- 路径参数经过一次严格的 URL percent-decoding；非法编码、解码后的 `/`、`\\`、`.` 或 `..` 均拒绝匹配。
- 构建快照时，核心为每条内容路由枚举结果并检查规范化 URL 的唯一性。两个内容记录产生同一 URL、一个内容路由命中多个记录，或必需参数无法生成时，构建失败。
- 一条内容记录可以被多个路由发布，但必须存在一个显式 `canonical: true` 的路由；`item.url` 始终取 canonical URL。只有一个路由时默认 canonical。
- 路由按声明顺序匹配，但静态段优先于参数段；完全等价或可能产生歧义的路由在启动或 reload 时拒绝加载。
- 查询参数不参与主题路由匹配。尾部斜杠策略由核心统一为“除 `/` 外不保留尾部斜杠”，不向主题暴露两种形式。

集合查询在 V1 只支持等值、`not`、包含、存在性过滤，以及一个或多个字段排序。类型不匹配是构建错误，不做隐式转换。排序始终在主题声明的字段之后追加 `id` 升序作为稳定的最终排序键。分页使用核心解析的正整数 `page` 查询参数；非法值返回 400，超过末页返回空列表而不是隐式跳转。

## 6. SSR 与 islands

普通页面请求流程：

```text
请求
→ 核心匹配主题路由
→ 读取已加载的内容快照
→ 执行声明式集合查询
→ 调用允许的插件服务
→ Preact SSR
→ 返回完整 HTML
```

文章页、列表、标签页默认不加载整站 JavaScript。

评论表单、站内搜索等交互块才作为 island 单独打包和 hydration：

```text
评论 Island
→ 核心 Action
→ 评论插件服务
→ SQLite
```

主题构建时，核心为每个 island 生成带内容哈希的浏览器资源和一份 island manifest。SSR 只输出 island 名称、可序列化 props 与资源引用；浏览器入口根据 manifest hydration。props 必须通过核心的 JSON 序列化校验，不允许函数、循环引用或隐式读取服务端状态。构建失败时服务拒绝启动，不使用新旧资源混合的 manifest。带内容哈希的资源使用长期不可变缓存；HTML 与 manifest 不使用长期缓存。

请求未匹配主题路由时返回 404，方法不受支持时返回 405。SSR 调用插件服务时由核心施加超时；服务失败或页面渲染异常时，不发送部分 HTML，统一返回带请求 ID 的 500 页面并记录服务名、耗时与服务端堆栈。动态请求失败不会使当前内容快照失效。

## 7. 插件：后端服务层

插件也是本地、完全可信的代码。V1 不对插件做沙箱或能力隔离，只加载用户明确配置的本地插件。

插件以提供后端服务为架构职责。下列能力限制描述的是核心支持的插件接口，不是针对恶意代码的安全边界。插件绕过接口修改其他插件的数据表、主题文件或核心状态属于未定义行为。

插件可以：

- 声明服务方法；
- 声明输入/输出 schema；
- 声明核心 Action；
- 声明 `remark` / `rehype` 插件，扩展 Markdown AST 转换和静态内容块渲染；
- 维护自己的 SQLite 表；
- 提供显式数据库迁移。

插件不可以：

- 注入页面布局、全局 HTML 或 CSS；
- 为 Markdown 内容块提供浏览器组件或 island；
- 定义主题路由；
- 让主题访问其数据表；
- 在 V1 订阅事件、启动后台任务或定时任务。

### Markdown 扩展

插件可以声明 `remark` / `rehype` 插件，为站点增加自定义 Markdown 语法和静态内容块渲染：

```ts
export default definePlugin({
  name: "callout",

  markdown: {
    remarkPlugins: [remarkDirective, remarkCallout],
    rehypePlugins: [rehypeCallout],
  },
})
```

例如插件可以将：

```md
:::callout{type="warning"}
这里是警告内容。
:::
```

转换为文章正文中的静态 HTML 片段。插件可以为内容片段生成语义化标签、class 与 `data-*` 属性，具体视觉样式由主题提供。

Markdown 处理顺序为：

```text
Markdown
→ 核心 remark 插件
→ 插件 remarkPlugins
→ remark-rehype
→ 插件 rehypePlugins
→ HTML
→ 内容快照
```

插件扩展按 `site.config.ts` 中的插件顺序注册。扩展发生异常时，本次内容快照构建失败；热重载场景下继续使用上一份有效快照。

V1 只支持静态 Markdown 内容块。需要浏览器交互的插件组件、插件 island 和插件前端资源构建协议不进入 V1。

评论插件示例：

```ts
export default definePlugin({
  name: "comments",

  services: {
    comments: {
      list(input: { contentId: string }) {},
      create(input: { contentId: string; body: string }) {},
    },
  },

  actions: {
    "comments.create": {
      service: "comments.create",
      public: true,
    },
  },
})
```

主题通过类型化服务契约读取评论；浏览器通过核心 Action 提交评论。插件私有实现、审核逻辑与数据库结构不会泄露给主题。

这里的“不泄露”指核心接口不会暴露插件私有实现，不是进程级访问隔离。主题若直接读取 SQLite 或导入插件内部文件，将失去兼容性保证。

### Action 基线

所有公开 Action 都经过核心统一入口，并默认具备以下约束：

- 只接受插件声明的 HTTP 方法和内容类型，默认只允许 `POST application/json`。
- 请求体默认上限为 64 KiB；插件可以声明更小的限制，不能自行提高全局上限。
- 输入与输出都必须通过插件声明的 schema；校验失败返回 400，不调用插件服务。
- 核心为每个 Action 设置执行超时，并把插件异常转换为不泄露堆栈和数据库信息的标准错误响应。
- 核心提供基于客户端地址与 Action 名称的进程内速率限制。它只适用于 V1 的单进程部署，不声称支持代理链或多实例。
- 使用 cookie 或其他浏览器凭证的 Action 必须启用 Origin 校验和 CSRF token；无凭证的公开 Action 仍执行 Origin 校验、速率限制与请求体限制。
- 核心生成请求 ID 并记录 Action 名称、状态码、耗时和限制命中情况，不记录评论正文等敏感请求体。
- 写操作不自动重试。需要防止重复提交的插件可以声明幂等键，并负责定义幂等结果。

评论插件的 `create` 还必须定义正文长度、默认审核状态以及内容不存在时的行为，才能作为 Action 与插件存储接口的验收样例。

插件配置放在 `site.config.ts`；动态数据放 SQLite。主题和插件包及其额外依赖由
站点所有者直接通过 Bun、`package.json` 和 `bun.lock` 管理，核心不提供扩展包的
安装、升级或删除命令。

```ts
export default defineSite({
  theme: "./themes/my-theme/theme.ts",
  plugins: ["@diitey/plugin-comments"],
})
```

每个插件必须声明稳定的插件 ID、插件版本与所需数据库 schema 版本。核心在 SQLite 中维护插件迁移记录，记录插件 ID、迁移 ID、校验和与执行时间。主题、插件、初始内容快照和 islands 校验成功后，核心在开始 HTTP 监听前用一个事务自动执行全部待处理插件迁移；任一迁移失败会整体回滚并终止启动。内容 `reload` 不执行插件迁移。

已执行插件迁移的校验和发生变化属于错误，不允许静默重跑。如果数据库版本高于插件声明支持的版本，服务拒绝启动。V1 不提供自动降级；需要回退时由插件提供显式向下迁移或从备份恢复。

## 8. 内容热重载

不监听文件变化。Obsidian、Git 或编辑器修改文件后，由用户手动执行：

```bash
diitey reload
```

流程：

```text
扫描 content/
→ 解析 Markdown 与 YAML
→ 校验 id、created 与主题 schema
→ 建立新的内容索引快照
→ 验证集合与路由
→ 成功：原子替换旧快照
→ 失败：保留旧快照并报告错误
```

因此线上请求永远只读取一份完整、一致的内容快照；编辑半成品不会影响网站。

快照替换以引用交换完成。请求开始时捕获当时的快照引用，并在整个请求期间继续使用它；reload 成功后新请求读取新快照，正在执行的请求可以正常完成。旧快照在没有请求引用后由运行时回收。

同一时刻只允许一个 reload 构建。已有 reload 运行时，新的请求返回“正在执行”状态，不再排队启动第二次扫描。构建具有可配置超时；超时、解析错误、主题校验错误或插件 Markdown 扩展异常都保留旧快照，并记录本次失败摘要。`status` 必须同时报告当前有效快照版本、最后成功时间、最近一次尝试结果和当前是否正在 reload。

内容热重载不重启 Bun 进程。主题、插件、核心代码或 `site.config.ts` 改动时才重启服务。

管理 CLI：

```bash
diitey start
diitey reload
diitey status
```

V1 使用只监听 loopback 地址的管理 HTTP 接口连接 CLI 与运行中服务。服务启动时在 `data/` 下写入仅当前用户可读的运行信息文件，其中包含管理端口、进程 ID 和随机管理 token；进程正常退出时删除该文件。CLI 读取文件后使用 Bearer token 调用管理接口。

管理接口不是稳定的公开 API，只服务同机 CLI。请求必须验证 token，并拒绝非 loopback 来源。运行信息文件缺失、进程 ID 已失效或 token 不匹配时，CLI 返回明确错误，不尝试启动新服务。`reload` 返回本次构建 ID；同步等待构建完成并输出结果，超时后可用 `status` 查询同一构建 ID。

## 9. 路由优先级

核心保留系统命名空间，主题不得覆盖：

```text
/_system/*   管理与状态，仅存在于 loopback 管理监听器
/_action/*   浏览器对插件服务的调用
/assets/*    主题与 island 静态资源
/*           主题路由
```

公开站点监听器收到 `/_system/*` 请求时始终返回 404。管理监听器只提供 `/_system/*`，不加载主题路由、Action 或公开静态资源。

## 10. V1 不做什么

明确不进入 V1：

- 内置账号、登录、用户后台；
- 文章网页编辑器；
- 文件自动监听；
- 多实例部署、共享文件系统；
- Serverless；
- 旧 URL、重定向与 SEO 迁移；
- Obsidian 双链/嵌入；
- 插件市场、远程下载安装；
- 后台任务、事件总线、定时任务；
- 插件 UI 注入。

## 11. 实施顺序

实施按可运行的纵向切片推进，每个切片都必须有自动化验收命令，不先按技术层铺开全部抽象。

1. **最小发布闭环**：建立 Bun CLI、`Bun.serve` 与配置加载；读取一个 Markdown 文件，校验 `id`/`created`，建立单记录快照，通过一个主题集合和一个路由完成 Preact SSR。验收结果是请求固定 URL 能得到包含标题和正文的完整 HTML。
2. **可靠 reload 闭环**：实现 loopback 管理接口、运行信息文件、不可变快照交换、并发控制、`diitey reload` 与 `diitey status`。验收必须覆盖成功替换、失败保留旧快照，以及 reload 期间的请求仍使用一致快照。
3. **完整内容与路由模型**：扩展到目录扫描、主题 schema、集合查询、稳定排序、分页、路由映射和 URL 冲突检查。验收必须覆盖重复 ID、重复 URL、嵌套路径、非法日期及确定性排序。
4. **Markdown 扩展闭环**：实现统一的 remark/rehype 管线和一个静态 callout 插件。验收必须覆盖插件顺序、转换异常导致构建失败，以及旧快照继续可用。
5. **island 闭环**：用 `Bun.build` 生成 island manifest、内容哈希资源、SSR props 和 hydration。验收必须确认普通文章不加载整站 JavaScript，只有声明的 island 被加载。
6. **动态行为闭环**：实现核心 Action 基线、SQLite 插件存储接口、版本检查和迁移记录；用评论插件完成“SSR 列表 + island 提交 + SQLite 写入”。验收必须覆盖输入限制、速率限制、插件异常、重复迁移和 schema 版本不兼容。
7. **发布前收口**：补齐错误页面、结构化日志、优雅退出、数据库备份说明、示例站点和端到端测试；冻结 V1 的主题接口、插件接口与 CLI 行为。

每个切片完成后先以当前接口运行代码审查，再进入下一个切片。只有在第二个真实调用方或替代实现出现时才提取新的公共接口，避免为假设中的扩展点提前增加浅层抽象。

这份方案的关键分工是：**内容由文件拥有，网站结构由主题拥有，动态行为由插件拥有，核心负责让它们一致地协作。主题和插件均为完全可信的本地代码。**
