# 插件发布资源与 document head 片段

## 状态

已接受

## 背景

插件可以通过 Markdown 扩展产生静态语义标记，但此前不能为发布页携带字体等浏览器资源，也不能让主题 document 使用插件提供的样式。直接开放任意最终 HTML 变换会让插件修改 body、document 结构和响应期 HTML，破坏主题拥有信息架构的边界。

现有内容资源管线已经提供内容寻址、SHA-256、磁盘缓存、原子发布和 HTTP 资源服务，但其 `ContentResourceBuilder` 绑定了 Markdown URL 解析、内容目录 containment、内容快照和 reload 生命周期，不能直接承担插件资源的领域职责。

## 决策

插件可以声明发布侧静态资源，并在站点程序编译时生成一个任意 HTML **head 片段**。核心将 head 片段插入最终 document 的 `</head>` 前；插件不得通过该能力修改 body、替换完整 document 或接管 HTTP 响应。

插件资源与内容资源共享底层 artifact store 能力（文件读取、哈希、临时文件、原子 rename、字节长度、MIME、缓存清理），但使用独立适配器和生命周期：

- 内容资源属于内容快照，路径位于 `/assets/content/...`，随成功 `reload` 更新；
- 插件资源属于站点程序，路径位于 `/assets/plugins/<plugin-id>/...`，随重启更新；
- 插件必须声明显式 plugin ID；资源路径必须位于插件入口目录的真实路径树内；
- 插件资源加入有效发布视图的资源 map，并使用长缓存 immutable 响应；
- 如果共享磁盘缓存，清理器必须同时保护当前内容资源和站点程序插件资源；V1 可以使用独立插件资源缓存目录。

head 片段在 `compileSiteProgram` 阶段、资源 URL 已固定后生成一次，并按 `site.config.ts` 的插件声明顺序组合。片段生成失败使站点程序编译失败；内容 `reload` 不重新执行插件 setup、资源构建或 head 生成。

head 片段允许任意 HTML，因为主题和插件是站点所有者明确选择的可信本地代码（ADR-0006）。这不是对不可信插件的安全隔离；它只是一个受支持的兼容性边界。核心仍固定注入位置和生命周期，避免把该能力扩展成任意最终 HTML 变换。

## 取舍

### 选择的方案

插件可以为字体、CSS、meta、预加载提示或其他 head 内容提供资源和 HTML 片段；主题不需要复制插件资源文件，也不需要把插件包路径暴露给浏览器。

### 放弃的方案

- **任意最终 HTML transform**：能力过宽，会让插件修改 body 和完整 document，并要求在静态预渲染与请求期渲染之间定义复杂的一致性规则。
- **让 `ContentResourceBuilder` 直接处理插件文件**：会把内容目录、内容快照和 reload 语义错误地施加到站点程序资源。
- **继续要求站点把字体复制到 `public/`**：可行但插件不可自包含，资源也失去插件包的版本绑定。

## 后果

- 主题仍拥有页面信息架构和 body 呈现；插件只能贡献 head 片段。
- 插件资源改变需要重启，符合站点程序固定和 reload pin 语义。
- 需要新增插件资源构建、有效发布视图 map、HTTP 路由和测试。
- 任意 head HTML 仍可能影响浏览器行为；这是可信本地扩展模型下明确接受的风险。
